## [2025-12-23] 下单分析表空值处理

### 🔄 主要变更

为下单分析表添加空值处理逻辑：如果"店铺"、"下单人"、"所属部门"字段为空，则自动填充为"无"。

### 📊 处理逻辑

```python
# 处理空值：如果字段为空则填充为"无"
shop = shop if shop and shop.strip() else '无'
orderer = orderer if orderer and orderer.strip() else '无'
department = department if department and department.strip() else '无'
```

### 💡 处理规则

| 字段 | 原值 | 处理后 |
|------|-----|--------|
| 店铺 | 空字符串 `""` | `"无"` |
| 店铺 | 空格 `"   "` | `"无"` |
| 店铺 | `None` 或不存在 | `"无"` |
| 下单人 | 空字符串/空格/None | `"无"` |
| 所属部门 | 空字符串/空格/None | `"无"` |

### ⚙️ 影响范围

**修改的文件**：
- `jobs/purchase_analysis/generate_analysis.py`
  - `merge_data()` 函数 - 在数据组装前添加空值检查

### 🎯 优势

1. **数据完整性**：避免数据库中出现空值
2. **查询友好**：便于SQL查询和统计
3. **显示清晰**：前端展示时不会出现空白
4. **统一标准**：所有空值统一显示为"无"

### 📋 示例

#### 之前
```python
{
    'SKU': 'ABC123',
    '店铺': '',           # 空字符串
    '下单人': '',         # 空字符串
    '所属部门': ''        # 空字符串
}
```

#### 现在
```python
{
    'SKU': 'ABC123',
    '店铺': '无',         # 自动填充
    '下单人': '无',       # 自动填充
    '所属部门': '无'      # 自动填充
}
```

## [2025-12-23] 下单分析表增量更新策略

### 🔄 主要变更

下单分析表改为**增量更新策略**：只更新**最近4个月 + 未来所有时间**的数据，保留4个月之前的历史数据。

### 📊 数据处理变化

#### 之前
- 时间限制：只处理前2个月+本月+未来3个月（共6个月）
- 数据库操作：每次 `TRUNCATE` 清空整表后重新插入
- 历史数据：6个月前的数据会被删除

#### 现在（增量更新）
- 时间范围：**最近4个月 + 未来所有时间**
- 数据库操作：
  1. ✅ 只删除4个月内及未来的旧数据
  2. ✅ 插入4个月内及未来的新数据
  3. ✅ **保留4个月之前的历史数据**
- 历史数据：永久保留

### 💡 更新逻辑

假设今天是 **2025-12-23**：

```python
# 1. 计算起始日期（4个月前的月初）
start_date = '2025-08-01'

# 2. 删除该时间段的旧数据
DELETE FROM `下单分析表` WHERE `日期` >= '2025-08-01'

# 3. 查询并插入新数据
SELECT ... FROM 采购单 WHERE DATE(创建时间) >= '2025-08-01'
SELECT ... FROM 运营下单表 WHERE DATE(下单时间) >= '2025-08-01'

# 4. 保留的历史数据
2025-08-01 之前的所有记录保持不变
```

### 📅 数据保留示例

| 日期区间 | 处理方式 | 说明 |
|---------|---------|-----|
| 2025-08-01 之前 | ✅ 保留 | 历史数据永久保留 |
| 2025-08-01 ~ 今天 | 🔄 更新 | 删除旧数据，插入新数据 |
| 今天之后 | 🔄 更新 | 包含未来计划的下单数据 |

### 💡 配合采购单增量更新

完整的数据更新策略：

| 表名 | 数据策略 | 说明 |
|------|---------|-----|
| **采购单表** | 增量更新（最近4个月） | 保留4个月前的历史数据 |
| **运营下单表** | 全量保留 | 从飞书获取后永久保存 |
| **下单分析表** | 增量更新（最近4个月+未来） | 保留4个月前的历史数据 |

**修改的文件**：

  - `save_to_database(data_list, start_date)` - 改为删除指定范围而非清空整表
  - `print_statistics(data_list, start_date)` - 更新日志输出

### 🚀 优势

4. **未来数据**：
   - 运营下单表可能包含未来的计划数据

## [2025-12-23] 采购单增量更新策略

### 🔄 主要变更

将采购单数据更新策略从**全量更新**改为**增量更新（最近4个月）**。

### 📊 更新策略变化

#### 之前（全量更新）
- 时间范围：从每年5月1日到当前日期
- 数据处理：每次删除全部数据后重新插入
- 问题：历史数据会被删除，处理时间长

#### 现在（增量更新）
- 时间范围：**最近4个月**（从4个月前的月初到当前日期）
- 数据处理：
  1. ✅ **删除**最近4个月的旧数据
  2. ✅ **插入**最近4个月的新数据
  3. ✅ **保留**4个月之前的历史数据
- 优势：
  - 保留历史数据
  - 更新速度更快
  - 减少数据库压力

### 💡 更新逻辑

```python
# 1. 计算日期范围（最近4个月）
end_date = datetime.now()
start_date = calculate_months_ago(4)  # 4个月前的月初

# 2. 删除该时间段的旧数据
DELETE FROM `采购单` 
WHERE DATE(`创建时间`) >= start_date 
  AND DATE(`创建时间`) <= end_date

# 3. 插入该时间段的新数据
INSERT INTO `采购单` (...)
```

### 📅 示例说明

假设今天是 **2025-12-23**：

- **start_date**: `2025-09-01`（4个月前的月初）
- **end_date**: `2025-12-23`（今天）
- **删除范围**: 2025-09-01 至 2025-12-23 的所有记录
- **获取数据**: 从API拉取 2025-09-01 至 2025-12-23 的采购单
- **保留数据**: 2025-09-01 之前的历史记录不受影响

### ⚙️ 影响范围

**修改的文件**：
- `jobs/purchase_analysis/fetch_purchase.py`
  - 新增 `calculate_months_ago()` 函数
  - 修改日期范围计算逻辑
  - 更新日志输出信息

**数据库变化**：
- ✅ 保留4个月之前的历史数据
- 🔄 更新最近4个月的数据
- ❌ 不再清空整个表

### 🚀 优势

1. **保留历史数据**：4个月前的数据不会被删除
2. **更新效率高**：只处理4个月的数据，速度更快
3. **支持回溯**：可以查看更早期的历史记录

### 📌 注意事项

1. **首次运行**：如果数据库为空，会自动创建并填充最近4个月的数据
2. **定期运行**：建议每天运行一次，保持数据最新
3. **历史数据**：如需更长时间的数据，需要手动调整 `calculate_months_ago(4)` 中的月份数

## [2025-12-23] 分析表维度优化

### 🔄 主要变更

将下单分析表的数据维度从**按月汇总**改为**按天汇总**。

### 📊 数据结构变化

#### 之前（按月）
- 维度：`SKU + 店铺 + 月份`
- 示例：`('ABC123', 'JQ-US', '2025-12')` → 汇总整个12月的数据

#### 现在（按天）
- 主维度：`SKU + 店铺 + 日期`
- 辅助字段：保留月份字段用于查询
- 示例：`('ABC123', 'JQ-US', '2025-12-23')` → 每天独立记录

### 📝 表结构变化

```sql
CREATE TABLE `下单分析表` (
    `id` INT AUTO_INCREMENT PRIMARY KEY,
    `SKU` VARCHAR(255),
    `店铺` VARCHAR(255),
    `面料` VARCHAR(255),
    `下单人` VARCHAR(255),
    `所属部门` VARCHAR(255),
    `日期` DATE COMMENT '下单日期',        -- 新增：主维度
    `月份` VARCHAR(50) COMMENT '所属月份',  -- 保留：辅助查询
    `实际已下单数量` INT,
    `预计下单数量` INT,
    `下单差值` INT,
    `更新时间` DATETIME,
    INDEX idx_sku_shop_date (SKU, 店铺, 日期),  -- 新增索引
    INDEX idx_month (月份),                      -- 新增索引
    INDEX idx_date (日期)                        -- 新增索引
)
```

### 💡 优势

1. **更细粒度的数据**：可以看到每天的下单情况
2. **保留月度统计**：通过月份字段仍可按月查询
3. **灵活查询**：
   - 按日查询：`WHERE 日期 = '2025-12-23'`
   - 按月查询：`WHERE 月份 = '2025-12'`
   - 按日期范围：`WHERE 日期 BETWEEN '2025-12-01' AND '2025-12-31'`

### 📋 查询示例

```sql
-- 查询某天的数据
SELECT * FROM 下单分析表 
WHERE 日期 = '2025-12-23' AND 店铺 = 'JQ-US';

-- 按月汇总
SELECT 
    SKU,
    店铺,
    月份,
    SUM(实际已下单数量) as 月度实际数量,
    SUM(预计下单数量) as 月度预计数量
FROM 下单分析表
WHERE 月份 = '2025-12'
GROUP BY SKU, 店铺, 月份;

-- 查询最近7天的数据
SELECT * FROM 下单分析表
WHERE 日期 >= DATE_SUB(CURDATE(), INTERVAL 7 DAY)
ORDER BY 日期 DESC, 店铺, SKU;

-- 按店铺统计本月每天的下单量
SELECT 
    店铺,
    日期,
    SUM(实际已下单数量) as 当日总数
FROM 下单分析表
WHERE 月份 = DATE_FORMAT(NOW(), '%Y-%m')
GROUP BY 店铺, 日期
ORDER BY 日期 DESC;
```
