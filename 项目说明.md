# pythondata 项目说明

基于 **领星 OpenAPI + MySQL + 飞书** 的数据采集、同步与分析管道。包含 Listing/利润报表/费用单同步、飞书多维表读写、采购与库存分析等能力。

## 二、各模块作用说明

### 1. common（公共层）

| 文件 | 作用 |
|------|------|
| `config.py` | 从项目根目录 `.env` 读取配置：环境、日志、领星 API、数据库、飞书 App ID/Secret、采集参数等。无 `.env` 时会提示复制 `config.example.env`。 |
| `database.py` | MySQL 连接与游标封装，供各 job 使用。 |
| `feishu.py` | 飞书 OpenAPI 客户端（鉴权、多维表读写等）。 |
| `logger.py` | 统一日志（按日期目录写入 `logs/`）。 |

### 2. lingxing（领星 SDK）

领星 OpenAPI 的 Python3 封装，包含签名、Token、HTTP 请求等。业务代码通过此 SDK 调用领星接口。详见 `lingxing/README.md`。

### 3. jobs/Sync_data（领星数据同步 + 利润/费用）

| 文件 | 作用 |
|------|------|
| `fetch_listing.py` | 从领星拉取亚马逊 Listing，按 sid + seller_sku 入库。 |
| `fetch_profit_report_msku_daily.py` | 拉取利润报表 MSKU 按日数据（dataDate+sid+msku+asin），写入利润报表表。 |
| `update_profit_report_calculated_fields.py` | 用产品管理、头程单价等更新利润报表中的计算字段，分阶段查询+批量更新。 |
| `create_fee_management.py` | 根据利润报表数据创建费用单（查费用类型、限流/重试）。 |
| `create_fee_management_daily.py` | 按日维度作废并重新创建费用单（默认本月）。 |
| `diagnose_query_performance.py` | 利润报表相关 SQL 性能诊断，不执行业务逻辑。 |

### 4. jobs/feishu（飞书多维表与业务）

| 文件 | 作用 |
|------|------|
| `Shipment_Number.py` | 领星货件数据 → 按 SPU+箱号聚合 → 写飞书「货件单号」多维表。 |
| `fetch_fabric_params.py` | 飞书「定制面料参数」→ 入库。 |
| `fetch_feishu_data.py` | 飞书面料表（款号、季节、面料等）→ 按款号+面料展开入库。 |
| `write_sales_to_feishu.py` | 数据库「销量统计_MSKU月度」→ 按店铺写飞书多维表（SKU+近 13 月销量）。 |
| `write_inventory_to_feishu.py` | 仓库库存明细 + 销量品名匹配 → 写飞书。 |
| `write_order_forecast_to_feishu.py` | 飞书多维表中运营填的「预计下单」→ 写入「运营预计下单」表。 |
| `generate_order_comparison.py` | 采购单 + 运营预计下单 → 按 SKU+店铺+月汇总 → 下单对比表（可写飞书）。 |
| `generate_fabric_forecast.py` | 运营预计下单 + 面料核价 → 面料用量预估。 |
| `update_product_images.py` | 按飞书 SKU 查库 listing 缩略图 → 回写飞书图片字段。 |

### 5. jobs/purchase_analysis（采购与库存分析）

| 文件 | 作用 |
|------|------|
| `main.py` | 采购分析流水线总入口：产品管理(7 天增量) → FBA 库存 → 仓库库存明细 → 销量月度 → 采购单 → 库存预估；可发飞书通知；部分步骤在代码中可配置禁用。 |
| `shop_mapping.py` | 领星店铺 ID ↔ 名称映射，供其他任务使用。 |
| `fetch_product.py` | 产品管理（领星，最近 7 天增量）。 |
| `fetch_fba_inventory.py` | FBA 库存（领星）。 |
| `fetch_inventory_details.py` | 仓库库存明细（领星）。 |
| `fetch_sale_stat_v2_msku_monthly.py` | 销量统计 MSKU 月度（领星）。 |
| `fetch_purchase.py` | 采购单（领星）。 |
| `generate_inventory_estimate.py` | FBA + 本地库存 → 按 SKU+店铺+库存状态生成库存预估表。 |

### 6. 根目录工具脚本

| 文件 | 作用 |
|------|------|
| `retry_failed_fee_orders.py` | 按失败批次从库中重新查数据，以 MSKU 维度逐个重试费用单。 |
| `resubmit_failed_mskus.py` | 对指定失败 MSKU 列表逐个重新提交费用单（dimension=1）；遇「MSKU 不存在」跳过。 |
| `recreate_all_fee_orders.py` | 重建全部费用单（按业务逻辑全量重跑）。 |
| `search_in_orders.py` | 订单检索辅助工具。 |

## 五、服务器定时任务（Cron）使用说明

本项目部署到 Linux 服务器后，通过**服务器上已有的**定时配置（如 `/etc/crontab` 里 hourly/daily/weekly/monthly 的 run-parts）自动执行。下面先说明服务器现有机制，再说明如何把本项目的脚本接进去。

### 5.1 当前在跑的定时脚本是什么、跑的内容

若已按 5.4 在 `/etc/cron.daily/` 里放了本项目的入口脚本（如 `pythondata-daily`），则**每天 06:25** 会由系统自动执行该入口，入口再**按顺序**调用下面 4 个脚本，每个脚本跑的内容如下：

| 顺序 | 脚本 | 跑的内容 |
|------|------|----------|
| 1 | `run_profit_report_fee_sync.sh` | ① 采集利润报表 MSKU 按日数据（本月 1 号至今） ② 更新利润报表计算字段；失败会发飞书并退出。 |
| 2 | `run_data_sync.sh` | ① 采集亚马逊 Listing ② 采集面料参数 ③ 采集飞书面料数据 ④ 生成下单对比表 ⑤ 生成面料预估表；任一步失败发飞书并退出。 |
| 3 | `run_jobs.sh` | 执行采购分析流水线：`jobs.purchase_analysis.main`（产品管理增量、FBA 库存、仓库库存明细、销量月度、采购单、库存预估）。 |
| 4 | `run_inventory_sync.sh` | ① 采集仓库库存明细 ② 写入飞书多维表 ③ 获取货件单号并写飞书；任一步失败发飞书并退出。 |

**说明**：当前**未**把「创建费用单」放进定时（`run_profit_report_fee_sync.sh` 里步骤 3 已注释），费用单若需要可单独在 Windows 用批处理或另加一条 cron。

### 5.2 服务器现有的 Cron 机制（已有配置）

服务器上**已经存在**的定时大致如下（以你提供的为例）：系统用 **run-parts** 按周期执行目录下的脚本（无扩展名的可执行文件会被执行）：

| 目录 | 执行时间（示例） | 说明 |
|------|------------------|------|
| `/etc/cron.hourly/` | 每小时第 17 分（`17 * * * *`） | 需要每小时跑的任务 |
| `/etc/cron.daily/` | 每天 06:25（`25 6 * * *`） | 需要每天跑的任务 |
| `/etc/cron.weekly/` | 每周日 06:47（`47 6 * * 7`） | 需要每周跑的任务 |
| `/etc/cron.monthly/` | 每月 1 号 06:52（`52 6 1 * *`） | 需要每月跑的任务 |

- 执行身份为 **root**，环境变量继承自系统（如需可配置 `PATH` 等）。
- 放在上述目录里的必须是**可执行**脚本（`chmod +x 脚本名`），且**不要带扩展名**（如不要用 `xxx.sh`），否则 run-parts 可能不执行。

### 5.3 项目在服务器上的约定

- **项目路径**：脚本内写死为 `/opt/apps/pythondata`，请将代码部署到该目录（或修改各 `scripts/run_*.sh` 中的 `PROJECT_DIR`）。
- **虚拟环境**：使用 `$PROJECT_DIR/venv`，即 `/opt/apps/pythondata/venv`，定时任务通过该 venv 的 Python 运行。
- **日志**：所有任务日志在项目下 `logs/`，按日期分目录，便于按天排查。

### 5.4 推荐：把本项目接到 cron.daily（你放到服务器后这样接）

在**服务器已有**的「每天 06:25 执行 /etc/cron.daily/」基础上，在 `/etc/cron.daily/` 里加一个**本项目的入口脚本**即可。推荐在 **每天** 跑的数据同步、利润报表、采购分析、库存同步都通过该入口调用本项目的 `scripts/run_*.sh`。

**步骤一：在 /etc/cron.daily 里增加入口脚本**

例如新建 `/etc/cron.daily/pythondata-daily`（无后缀、可执行）：

```bash
sudo touch /etc/cron.daily/pythondata-daily
sudo chmod +x /etc/cron.daily/pythondata-daily
sudo nano /etc/cron.daily/pythondata-daily
```

**步骤二：在入口脚本里按顺序调用本项目（示例）**

以下为示例内容，可按需要删减或调整顺序；失败时脚本会退出，便于在日志或飞书里发现：

```bash
#!/bin/bash
# pythondata 每日定时任务入口（由 run-parts 在每天 06:25 执行）
PROJECT_DIR="/opt/apps/pythondata"

# 1. 利润报表 + 更新计算字段（不创建费用单时可只跑前两步）
"$PROJECT_DIR/scripts/run_profit_report_fee_sync.sh" || true

# 2. 数据同步：Listing、面料、下单对比、面料预估
"$PROJECT_DIR/scripts/run_data_sync.sh" || true

# 3. 采购分析流水线
"$PROJECT_DIR/scripts/run_jobs.sh" || true

# 4. 库存同步：库存明细 → 飞书 → 货件单号
"$PROJECT_DIR/scripts/run_inventory_sync.sh" || true
```

使用 `|| true` 可避免某一步失败导致后续不执行；若希望某一步失败就整体失败，可去掉对应行的 `|| true`。

### 5.5 可选：用用户 crontab 指定时间

若希望**不用 run-parts**，而是自己定具体时间，可用 root 或部署用户的 crontab：

```bash
sudo crontab -e -u root
```

示例（每天 6:30 跑利润报表，7:00 跑数据同步，7:30 跑采购分析，8:00 跑库存同步）：

```cron
30 6 * * * /opt/apps/pythondata/scripts/run_profit_report_fee_sync.sh >> /opt/apps/pythondata/logs/cron_profit.log 2>&1
0  7 * * * /opt/apps/pythondata/scripts/run_data_sync.sh >> /opt/apps/pythondata/logs/cron_data_sync.log 2>&1
30 7 * * * /opt/apps/pythondata/scripts/run_jobs.sh >> /opt/apps/pythondata/logs/cron_jobs.log 2>&1
0  8 * * * /opt/apps/pythondata/scripts/run_inventory_sync.sh >> /opt/apps/pythondata/logs/cron_inventory_sync.log 2>&1
```

脚本内已包含 `cd $PROJECT_DIR` 和虚拟环境激活，无需在 crontab 里再写 `cd` 或 `source venv`。

### 5.6 查看当前定时任务

```bash
# 当前用户的 crontab
crontab -l

# root 的 crontab（系统 run-parts 由 root 执行）
sudo crontab -l -u root

# 系统级配置（你提供的 hourly/daily/weekly/monthly 即在此或 cron.d）
cat /etc/crontab
ls -la /etc/cron.d/
ls -la /etc/cron.daily/
```

### 5.7 任务与周期建议汇总

| 任务 | 推荐周期 | 说明 |
|------|----------|------|
| 利润报表 + 更新计算字段 | 每日 | 领星利润数据按日更新，适合每天 06:25 或自定义时间 |
| 数据同步（Listing/面料/下单对比/面料预估） | 每日 | 保证飞书与库表一致 |
| 采购分析（main） | 每日 | 产品、FBA、库存、销量、采购、库存预估 |
| 库存同步（库存→飞书→货件单号） | 每日 | 飞书库存与货件信息更新 |
| 创建费用单（create_fee_management） | 按业务需要 | 若启用，可用 Windows 批处理或单独加一条每日 cron |

**总结**：服务器上已有的是「每日 06:25 执行 `/etc/cron.daily/` 下所有脚本」；你放到服务器跑的是本项目（pythondata）。只要在 `/etc/cron.daily/` 里增加一个入口脚本（如 5.4 的 `pythondata-daily`），在该脚本里调用本项目的 `scripts/run_*.sh`，即可在现有定时下每日自动跑上述任务。

## 六、飞书销量预估表与预估下单表相关代码及启动命令

### 6.0 相关代码是哪几个

| 业务 | 代码文件 | 作用 |
|------|----------|------|
| **飞书销量预估表**（库 → 飞书） | `jobs/feishu/write_sales_to_feishu.py` | 从数据库读「销量统计_MSKU月度」+「库存预估表」+「运营预计下单表」，按店铺生成/更新飞书多维表（含历史销量、预计销量、预计下单量等），即**写出**飞书销量预估表。 |
| **预估下单表**（飞书 → 库） | `jobs/feishu/write_order_forecast_to_feishu.py` | 从飞书多维表读取运营填写的「预计下单量」，写入数据库「运营预计下单表」，即**收集**预估下单内容到库。 |

说明：销量预估表的数据是 `write_sales_to_feishu.py` 从库写到飞书的；预估下单表的内容是 `write_order_forecast_to_feishu.py` 从飞书收到库的。两者使用同一套飞书多维表（先由 write_sales 写出，运营在表里填预计下单，再由 write_order_forecast 读回写库）。

### 6.0.1 各 .sh 在服务器上的启动命令（项目根为 /opt/apps/pythondata）

在服务器上需先进入项目目录并激活虚拟环境（若由 cron 调用，脚本内部会完成）。以下命令均可在项目根目录直接执行：
